API REFERENCE · v1

statehash.io API

A tiny HTTP API for anchoring arbitrary JSON on Base and looking the anchors up later. No SDK. No on-chain contract calls. Just JSON in, receipts out.

Base mainnet · 8453 Preview · subject to change v1

Introduction

statehash.io produces cryptographic receipts for JSON payloads. Each receipt is an anchor: a deterministic hash of your payload, submitted on-chain as a Base transaction, and stored with a namespace + external reference so you can look it up again later.

The API is intentionally small. There are exactly three verbs: anchor, fetch by ID, and fetch by namespace/external_ref. Anything else — verification, audit trails, long-term retention — happens off our servers, using public chain data.

Quickstart

  1. Request an API key. You'll be assigned a namespace.
  2. POST your first payload to /v1/anchors.
  3. Inspect the response. Follow the basescan link. You're done.
# Smallest possible anchor call
curl -X POST https://api.statehash.io/v1/anchors \
  -H "X-Api-Key: $STATEHASH_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"namespace":"demo","external_ref":"hello","payload":{"msg":"hello world"}}'

Base URL

https://api.statehash.io

A testnet endpoint (https://api.sepolia.statehash.io) writes to Base Sepolia. Use it from non-production environments so your demo traffic doesn't land on mainnet.

Authentication

Every request must include an API key. Either header works: 

X-Api-Key: sk_live_...
or
Authorization: Bearer sk_live_...

API keys are scoped to a single namespace. A key can only read and write anchors under its own namespace; cross-namespace access is never allowed, even with a valid key.

Namespaces

A namespace is a logical tenant — typically one per product or environment (e.g. acme-predictions, acme-predictions-staging). Namespaces are issued by us at key creation and cannot be reassigned.

Within a namespace, (namespace, external_ref) is unique: posting the same external_ref twice returns the existing anchor instead of creating a new one. This makes the API idempotent for retries.

Canonicalization

Two JSON blobs that look the same but differ in key order or whitespace must produce the same hash. We use a small deterministic serializer with the following rules:

  • Object keys are sorted lexicographically, recursively.
  • No extra whitespace. No trailing newlines.
  • Numbers are emitted with JavaScript's default Number format. Integers and floats that round-trip through JSON are safe.
  • null, true, false are literal.
  • Arrays preserve their order.
  • Strings are UTF-8 encoded.

The resulting canonical UTF-8 bytes are hashed with keccak256. Any language with a keccak implementation can reproduce it.

Commitment hash

The commitment_hash is a 0x-prefixed, lowercase hex string. It is a 32-byte keccak256 digest of the canonical JSON. It is the exact bytes written as calldata to the on-chain transaction. 

// Reference TypeScript
import { keccak256, stringToBytes } from "viem";

const commitment = keccak256(stringToBytes(canonicalJson(payload)));

Anchor lifecycle

  1. accepted — request received, payload canonicalized, row inserted in our DB.
  2. pending — transaction broadcast to Base but not yet mined.
  3. confirmed — transaction mined. tx_hash, block_number are populated. This is the terminal success state.
  4. failed — submission failed. You'll see an error field. Safe to retry with the same external_ref.

In v1, POST /v1/anchors is synchronous: it returns once the anchor reaches confirmed. Async mode is on the roadmap.

What is an agent?

An agent is a named actor under your namespace that emits claims — an AI model, an analyst team, a content publisher, a compliance reviewer. Every agent is provisioned with its own on-chain wallet. When you anchor a payload "as" an agent, the transaction is signed by that agent's wallet, not a shared one.

This is the primitive that makes an agent's track record independently auditable. A third party can point a Base RPC at the agent's address, enumerate every transaction, and hash every payload themselves — without ever calling statehash.io.

Agents are optional. If you omit agent_id on POST /v1/anchors, the anchor is signed by the shared system signer — the v0 behaviour. Use agents when you want per-actor enumeration on-chain.

Provision an agent

POST /v1/agents

Generates a fresh secp256k1 keypair, envelope-encrypts the private key, and returns the agent with its on-chain address. The private key never leaves the server.

Body parameters

FieldTypeDescription
namespace string Optional. Defaults to the namespace bound to your API key; if supplied it must match.
display_name string Optional stable label (e.g. sniper-v7). Unique within a namespace. Convenient for humans; the ULID id is the canonical handle.

Example request

curl -X POST https://api.statehash.io/v1/agents \
  -H "X-Api-Key: $STATEHASH_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"display_name": "sniper-v7"}'

Fetch an agent

GET /v1/agents/:id

Public endpoint — no API key required. Returns the agent record with live anchor count and activity window. This is the URL you hand out so anyone can audit an agent's history. 

curl https://api.statehash.io/v1/agents/agt_01KPD5CTRE7HR9W5PXN6MPWJ2R

Resolve an agent by wallet address

GET /v1/agents/by-address/:address

Public endpoint — no API key required. Given a 0x-prefixed Ethereum address, returns the agent it belongs to (or 404 if the address is not one of ours). Useful when you have a basescan URL in hand and want to cross-reference the agent id.

Why wallet-per-agent

Without per-agent wallets, nothing on-chain distinguishes one customer's anchors from another's — they all come from the same system signer. That's fine for integrity of any single anchor, but it breaks down when a verifier needs to ask "how many predictions has this agent ever made?". They'd have to trust us.

With per-agent wallets, a verifier can answer that question directly:

  1. Look up the agent: GET /v1/agents/:id → get address.
  2. Query Base for all transactions sent from that address.
  3. For each tx, decode the calldata — it's the raw commitment hash.
  4. Compare against the payloads the agent claims to have published.

If the numbers match, the agent has no hidden anchors. If the on-chain count exceeds the claimed count, some predictions were suppressed — selective disclosure detected.

Limitations. Per-agent wallets expose an agent's full on-chain count. They do not, by themselves, prevent an operator from running a second "shadow" agent and only advertising the good one. Bind each agent_id to a public identity (your domain, a signed X post, a registry) to close that gap.

Create an anchor

POST /v1/anchors

Body parameters

FieldTypeDescription
namespacerequired string Must match the namespace bound to your API key.
external_refrequired string Your ID for this anchor. Unique within the namespace; repeat calls return the existing anchor (idempotent).
payloadrequired any JSON Arbitrary JSON. Object, array, string, number, boolean, or null. Schema is entirely up to you.
agent_id string Optional. If present, the tx is signed by that agent's wallet. The agent must belong to the caller's namespace. See Agents.
store_payload boolean · default true If false, we keep only the hash and metadata. Use this for PII-heavy payloads.
metadata object Optional free-form metadata stored alongside the anchor (not hashed, not on-chain).

Example request

curl -X POST https://api.statehash.io/v1/anchors \
  -H "X-Api-Key: $STATEHASH_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "namespace": "acme-predictions",
    "external_ref": "game-42:home-win",
    "payload": {
      "match_id": "nba-2026-04-15-BOS-LAL",
      "prediction": { "home_win": 0.63 }
    }
  }'

Example response

{
  "id": "anc_01JSA7K2G9Q4M8Y3H2B8D5R4Z0",
  "namespace": "acme-predictions",
  "external_ref": "game-42:home-win",
  "commitment_hash": "0x7b0a4c…e91f3d",
  "chain_id": 8453,
  "tx_hash": "0x9f2a…c4d1",
  "block_number": 18421003,
  "status": "confirmed",
  "explorer_url": "https://basescan.org/tx/0x9f2a…",
  "created_at": "2026-04-15T19:02:14Z"
}

Fetch by ID

GET /v1/anchors/:id

Returns the anchor object for the given id. The anchor must belong to your namespace. 

curl https://api.statehash.io/v1/anchors/anc_01JSA7K2G9Q4M8Y3H2B8D5R4Z0 \
  -H "X-Api-Key: $STATEHASH_API_KEY"

Fetch by external reference

GET /v1/anchors/by-ref/:external_ref

Returns the anchor under the caller's namespace whose external_ref matches. Because (namespace, external_ref) is unique, this is always a single-result lookup. 

curl https://api.statehash.io/v1/anchors/by-ref/game-42:home-win \
  -H "X-Api-Key: $STATEHASH_API_KEY"

Independent verification

The whole point of statehash.io is that your users don't have to trust us. Given the original payload and the tx_hash, anyone can verify the anchor in about ten lines of code: 

import { createPublicClient, http, keccak256, stringToBytes } from "viem";
import { base } from "viem/chains";

const client = createPublicClient({ chain: base, transport: http() });
const tx = await client.getTransaction({ hash: txHash });

const expected = keccak256(stringToBytes(canonicalJson(payload)));
const onChain = tx.input; // 0x-prefixed calldata

console.log(expected === onChain ? "✓ anchored" : "✗ tampered");

You do not need statehash.io to be online for this to work. The RPC node serves the transaction; you do the hashing locally.

Errors

Errors are returned as JSON with a predictable shape:

{
  "error": {
    "code": "namespace_mismatch",
    "message": "Namespace does not match API key."
  }
}
HTTPCodeWhen
400invalid_requestMalformed JSON or missing required field.
401missing_api_keyNo API key header provided.
403invalid_api_keyKey not recognized.
403namespace_mismatchKey is not authorized for that namespace.
403agent_namespace_mismatchagent_id belongs to a different namespace than the caller.
404not_foundAnchor does not exist under this namespace.
404agent_not_foundagent_id does not exist.
409external_ref_conflictexternal_ref exists with a different payload hash.
409display_name_takenAgent display_name already exists in namespace.
500internal_errorSomething on our side. Retry-safe.

Rate limits

Preview accounts are limited to 30 anchors/minute and 10 concurrent requests. Limits are per API key. Higher limits are available on request.
