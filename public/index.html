The problem

Trust is asymmetric. Until it's on-chain.

If your app emits claims — predictions, model outputs, scores, receipts — anyone can accuse you of rewriting them after the fact. You can't easily prove you didn't. Logs are editable. Databases are editable. A screenshot is not evidence.

Without statehash.io. Your service says "the model predicted X on Tuesday." The user says "no it didn't." You show them your database. They shrug. You have no independent witness.
With statehash.io. Your service anchored the JSON on Tuesday. The payload's hash is in a Base transaction you don't control. Anyone can recompute the hash and confirm it. Dispute closed.

How it works

Three steps. No SDK.

You send JSON. We canonicalize it, hash it, and write the hash into a transaction on Base. You get an ID back. Everything else — the verification, the audit trail, the paper trail — is public.

01 / POST

Send your payload

POST /v1/anchors with your API key and any JSON. A namespace and external_ref let you look it up later.

02 / Canonicalize & hash

Deterministic commitment

We serialize keys in sorted order, strip whitespace, and take keccak256 of the bytes. Same JSON always produces the same 32-byte hash.

03 / Anchor on Base

Permanent on-chain record

The hash is submitted as calldata on Base. You get back tx_hash and a basescan link. Anyone can verify it by recomputing — no statehash.io account required.

Live example

One request. One permanent receipt.

Below is the full round-trip for anchoring a prediction. The response is small by design — the JSON lives with you; only the hash goes on-chain.

request.sh 
# Anchor any JSON. Schema is up to you.
curl -X POST https://api.statehash.io/v1/anchors \
  -H "X-Api-Key: $STATEHASH_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "namespace": "acme-predictions",
    "external_ref": "game-42:home-win",
    "payload": {
      "match_id": "nba-2026-04-15-BOS-LAL",
      "model": "forecaster-v7",
      "prediction": { "home_win": 0.63 },
      "emitted_at": "2026-04-15T19:02:11Z"
    }
  }'
response.json · 200 OK 
{
  "id": "anc_01JSA7K2G9Q4M8Y3H2B8D5R4Z0",
  "namespace": "acme-predictions",
  "external_ref": "game-42:home-win",
  "commitment_hash":
    "0x7b0a4c…e91f3d",
  "chain_id": 8453,
  "tx_hash": "0x9f2a…c4d1",
  "block_number": 18_421_003,
  "status": "confirmed",
  "explorer_url":
    "https://basescan.org/tx/0x9f2a…",
  "created_at": "2026-04-15T19:02:14Z"
}
Verify without us. Recompute keccak256 of the canonical JSON, then load the tx_hash from basescan and compare the calldata. If the bytes match, the claim is authentic. statehash.io is convenient, but not trusted — the chain is.

FAQ

Good questions.

What if statehash.io disappears?

The hashes are on Base. They do not depend on us. As long as you've retained the original JSON, you can verify any anchor by recomputing the hash and matching it against the on-chain transaction — even if this service is offline.

Can I store PII or secrets in the payload?

Don't. Only the 32-byte hash is on-chain, but by default we also store the canonical JSON server-side for lookup. You can opt out with store_payload: false and keep the raw payload yourself.

What's a "canonical" hash?

We sort object keys recursively, remove whitespace, and hash the resulting UTF-8 bytes with keccak256. It guarantees { "a": 1, "b": 2 } and { "b": 2, "a": 1 } produce the same commitment. The algorithm is stable and documented.

Which chains are supported?

Base (mainnet) and Base Sepolia (test) today. Additional chains are on the roadmap once there is demand.

Is this open-source?

The canonicalization and verification are specified openly so you can verify without trusting us. The server itself is in preview — contact us if you need a self-hosted build.
